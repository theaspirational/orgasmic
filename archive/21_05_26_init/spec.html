orgasmic — initial design spec

theaspirational · 2026-05-21

1 Deprecation notice

This file is historical provenance for the initial design session. It is no longer the active source of truth for implementation work.

Current project state lives in:

  • .orgasmic/project.org
  • .orgasmic/sprint.org
  • .orgasmic/decisions.org
  • .orgasmic/architecture.org
  • .orgasmic/glossary.org
  • docs/adr/*.org

When this archive conflicts with current code or current .orgasmic/ graph state, prefer the current code and graph, then update the graph/tasks/ADRs.

2 Mission

orgasmic is a repo-cloned, file-native, agent-aware coordination app for any developer running AI agents on their projects.

It is inspired by HAR (typed worker drivers, materialized views, structured event log) and org-mode (plain-text, Git-native, append-only project memory).

The "sweet spot" between the two: HAR's worker execution + manager pattern + materialized index, applied over .org files as the canonical substrate, with each developer running their own instance and multi-human coordination happening via plain Git on each project's `.orgasmic/` directory.

3 Install and data model

3.1 Two-layer layout

Modeled on hermes-agent. The cloned source lives inside a home directory, not as it — so updates never collide with user data.

$ORGASMIC_HOME = $HOME/.orgasmic       (overridable via env var)

$ORGASMIC_HOME/
  orgasmic/          ← git clone of this repo. Never edit for personal use.
    crates/
      orgasmic-core/
      orgasmic-daemon/
      orgasmic-cli/
      orgasmic-drivers/
    ui/
    shipped/
      agent-templates/*.org
      prompts/<kind>/<state>.org
      skills/native/<id>/SKILL.md
      schema/{tx,state-machine,slots,agent-template,prompt-template}.org
      manager/persona.org
      conventions/code-markers.org    ← advisory in-code marker format (v0.0.5)
      project-scaffold/{project,sprint,backlog,archive}.org
    scripts/install.sh
    target/release/orgasmic       ← built binary

  user/             ← user overrides + user-authored content. Gitignore-free
                      because it's not in the repo.
    agent-templates/*.org        ← overrides + user-authored
    prompts/<kind>/<state>.org   ← overrides
    skills/*.org                 ← skill metadata sidecars
    skills/native/<id>/SKILL.md  ← user-authored native skills
    manager/persona.org          ← if user overrides
    board.org                    ← projects board (pointer list)
    auth.org                     ← API tokens (gitignored at any commit)
    glossary.org                 ← global vocabulary (v0.0.5)
    decisions.org                ← global decision graph (v0.0.5)
    architecture.org             ← global architecture nodes (v0.0.5)
    context-map.org              ← multi-context map for global scope (v0.0.6)
    docs/adr/*.org               ← daemon-generated global ADRs (v0.0.6)

  state/
    tx/YYYY-MM.org               ← instance-scoped audit log (NOT committed)
    snapshots/<uuidv7>/          ← global-scope graph snapshots (v0.0.5)
  sessions/<run-id>.jsonl        ← per-run transcripts
  secrets/                       ← model provider API keys, plaintext files
  logs/orgasmic.log               ← daemon process log (rotates at 100MB)
  bin/orgasmic                    ← symlink to orgasmic/target/release/orgasmic
  config.yaml                    ← top-level user config

3.2 Loader rule (universal)

For every shipped-overridable file: read user/<relative-path> first; if absent, fall back to orgasmic/shipped/<relative-path>. Users never edit inside orgasmic/; if they want to customize, they copy to user/ and edit there.

3.3 Install one-liner

curl -fsSL https://raw.githubusercontent.com/<owner>/orgasmic/main/scripts/install.sh | bash

Installer clones the repo into $ORGASMIC_HOME/orgasmic/, runs cargo build --release, symlinks the binary into $ORGASMIC_HOME/bin/, and scaffolds the home dirs. Implementation in scripts/install.sh mirrors hermes-agent's shape (SSH-first clone, HTTPS fallback, auto-stash + restore on update, --branch, --dir, --no-build flags).

3.4 Update

orgasmic update does git fetch && git checkout BRANCH && git pull --ff-only inside orgasmic/, then cargo build --release, then refreshes the symlink. Auto-stashes any local edits inside orgasmic/ (which shouldn't exist if the override rule is followed) and prompts to restore.

3.5 Contributing

Standard fork + PR per CONTRIBUTING.md. Priorities: bug fixes > cross-platform > security > performance > shipped content > docs.

3.6 Three-tier shipped/optional/hub split

Applies to skills, agent templates, AND prompt templates:

  • Bundled: ships in orgasmic/shipped/, always available.
  • Optional: ships in orgasmic/optional/, opt-in via orgasmic {skills,agent-templates,prompts} install <id>.
  • Hub: community-published, installable by URL.

4 File substrate

4.1 Write surface

Hybrid by file role.

Path Edit model Writers
user/agent-templates/*.org direct rewrite human, manager (via RPC + flock)
user/prompts/<kind>/<state>.org direct rewrite human, manager
user/skills/*.org direct rewrite human, manager, discoverer
user/manager/persona.org direct rewrite human only
user/board.org direct rewrite human, manager
user/{glossary,decisions,architecture}.org RPC-only daemon (griller/architector/glossarist)
user/context-map.org direct rewrite human
user/docs/adr/*.org regenerated daemon (hand-edits rejected)
<project>/.orgasmic/*.org direct rewrite human, manager, worker (own task)
<project>/.orgasmic/{glossary,decisions,architecture}.org RPC-only daemon (griller/architector/glossarist)
<project>/.orgasmic/context-map.org direct rewrite human
<project>/docs/adr/*.org regenerated daemon (hand-edits rejected)
state/tx/YYYY-MM.org append-only daemon (single writer)
state/snapshots/<id>/*.org copy-on-snapshot daemon (atomic rename)
<project>/.orgasmic/tx/YYYY-MM.org append-only daemon (single writer)
<project>/.orgasmic/snapshots/<id>/*.org copy-on-snapshot daemon (atomic rename)
sessions/<run-id>.jsonl append-only per-run writer (daemon-mediated)

Workers may write directly to ** Evidence and ** Worklog inside their own claimed task heading, and may flip their own :STATE: via the transition_state(new, reason) tool. Everything else goes through daemon RPC.

4.2 Concurrency

Direct-edit files use advisory flock. Manager retries with timeout (default 30s); on timeout, posts a user-visible notification rather than overriding. Workers only contend on their own claimed task.

4.3 Parse failures

Daemon keeps last-good in-memory index. UI shows red badge with file + line number. Daemon does not crash on parse errors of working files. Tx files are the exception: daemon refuses to start if it detects post-hoc modification of historical tx (verified via manifest hash).

5 File formats

5.1 Tx files (.org, property-drawer-only)

Each tx is a heading + property drawer. No EDN blocks. No body content.

* TX 2026-05-21 14:32 run.start abc-123
:PROPERTIES:
:TX_ID:       tx-20260521-001
:TIME:        [2026-05-21 Thu 14:32:07]
:TYPE:        run.start
:ACTOR:       theaspirational
:MACHINE:     macbook-pro
:RUN:         abc-123
:TASK:        TASK-024
:WORKER:      worker-backend-03
:AGENT_TEMPLATE: implementer-claude
:PROMPT_HASH: sha256:f3a2…
:SESSION:     sessions/abc-123.jsonl
:END:

Rules:

  • Header line = timestamp + type + entity (so grep is the operational debug tool).
  • Properties typed by name; schema in shipped/schema/tx.org.
  • Pointers, not content (:SESSION:, :PAYLOAD:, :DIFF: → external refs).
  • Append-only via OAPPEND.
  • Daemon is the sole writer.
  • If a tx would need >12 properties or nested structure, refactor into a working-file rewrite — not a tx.

Two locations:

  • $ORGASMIC_HOME/state/tx/YYYY-MM.org: instance-scoped daemon audit log. Not committed. Per Q16-2.
  • <project>/.orgasmic/tx/YYYY-MM.org: project-scoped tx. Committed — teammates see the history after git pull.

5.2 Session files (JSONL)

One file per run: sessions/<run-id>.jsonl. Line 0 = envelope; subsequent lines are events.

{"schema":"orgasmic.session.v1","run":"abc-123","task":"TASK-024","worker":"...","agent_template":"...","started_at":"2026-05-21T14:32:07Z"}
{"t":"...","kind":"prompt.system","content":"..."}
{"t":"...","kind":"prompt.user","content":"..."}
{"t":"...","kind":"tool.call","tool":"Read","args":{"path":"src/foo.rs"}}
{"t":"...","kind":"tool.result","tool":"Read","ok":true,"bytes":1432}
{"t":"...","kind":"text.delta","content":"Looking at…"}
{"t":"...","kind":"finish","reason":"end_turn"}

Rules:

  • Every event has t + kind; the rest is kind-specific.
  • Format mirrors the worker's native transcript shape where possible (Claude Code session.jsonl, codex events, Anthropic Messages, etc.).
  • Append-only via OAPPEND.
  • No automatic compression in v0.0.1. Inspection stays one tail + jq away. Compression sweep can come later when disk becomes a problem.

5.3 No EDN anywhere

The migration spec for HAR (../har/docs/future/tx-log-to-org-hybrid-migration-spec.org) leaned on EDN payloads. orgasmic does not. If structured nested data is ever required, it gets introduced explicitly with a schema versioning bump — not as a casual escape hatch.

6 Domain model

6.1 Skills (Q6)

Orgasmic owns metadata, not skill content.

user/skills/discovered.org   ← auto-maintained index of all known skills
user/skills/<skill-id>.org   ← per-skill metadata sidecar

A sidecar:

* SKILL git-workflows
:PROPERTIES:
:ID:                git-workflows
:SOURCE:            ~/.claude/skills/git-workflows         ← absolute path
:KIND:              external                              ← external | native
:HOST:              claude-code                           ← claude-code | codex | hermes | agentskills | native
:VERSION:           2.1.0
:DESCRIPTION:       (extracted from SKILL.md frontmatter)
:TRIGGERS:          git push, rebase, hook failure
:FAVORITE:          true
:CATEGORY:          dev-workflow
:TAGS:              git, branching, rebase
:LINKED_TEMPLATES:  backend-implementer-v2, qa-reviewer-v1
:DISCOVERED_AT:     [2026-05-20 Wed 11:00]
:LAST_USED:         [2026-05-21 Thu 14:32]
:END:

** Notes
[user-editable]

** Manual overrides
:PROPERTIES:
:OVERRIDE_DESCRIPTION: Use only on branches that already passed CI.
:END:

Discovery (orgasmic skills discover) scans configured host roots (~/.claude/skills/, ~/.codex/skills/, ~/.hermes/skills/, agentskills.io cache, user/skills/native/) and writes/updates sidecars. Discovery is opt-in / on-demand at v0.0.1.

6.2 Agent templates (Q7)

Singular file per template; persona + linked skills + driver/model defaults + operational metadata.

* AGENT-TEMPLATE backend-implementer-v2
:PROPERTIES:
:ID:                          backend-implementer-v2
:KIND:                        implementer
:DEFAULT_DRIVER:              claude-acp
:DEFAULT_PROVIDER:            anthropic
:DEFAULT_MODEL:               claude-sonnet-4-6
:DEFAULT_EFFORT:              high
:LINKED_SKILLS:               git-workflows, rust-error-handling, sqlite-migrations
:APPLICABLE_STATES:           claimed, analyzing, implementing, fixing
:MAX_ITERATIONS:              20
:CONTEXT_BUDGET:              150000
:BABYSITTER_AGENT_TEMPLATE:   babysitter-stall-detector
:VERSION:                     2
:END:

** Persona
[short prose; injected as system content]

** Operating Rules
[bullet list; injected as additional system content]

** Notes
[free-form, user-editable; NOT injected]

:KIND: taxonomy (fixed in schema, custom as escape hatch): implementer | reviewer | planner | analyzer | babysitter | griller | architector | glossarist | custom | manager.

The griller, architector, and glossarist kinds belong to the post-v0.0.1 design-graph layer; see * Design-graph layer for their state machines and tool sets. planner and implementer (already in v0.0.1) gain new responsibilities in that layer: planner owns task proposal from settled architecture; implementer continues to own code edits. There is no reconciler kind — reconciliation is the manager re-running this same pipeline against the stale subgraph.

Babysitters are first-class agent templates with :KIND: babysitter, not a separate concept. Same machinery, different persona and state taxonomy.

If a linked skill is missing at compile time → compile error in dry-run, spawn blocked in production with a UI-visible "missing skill X for template Y."

6.3 Prompt templates (Q8)

One file per (kind, state) tuple.

shipped/prompts/<kind>/<state>.org
user/prompts/<kind>/<state>.org    ← overrides

Each file:

* PROMPT implementer/implementing
:PROPERTIES:
:KIND:    implementer
:STATE:   implementing
:VERSION: 2
:END:

** Spawn-time injection
[used when a run starts in this state; rare]

** Continuation injection
[used when a run transitions /into/ this state mid-run; common]

** Notes
[user-editable, not injected]

Slot syntax: /strict/ variable substitution, /no/ conditionals/loops. Registry in shipped/schema/slots.org. See that file for the v0.0.1 slot set. Dry-run validates and reports unknown / unfilled slots.

6.4 Skill injection (Q8.5)

={{skills.all}}= renders a manifest, not the bodies:

You have access to these skills. Use your Read tool to load any that match
your current situation:

- git-workflows
  Description: …
  Triggers: …
  Path: /Users/.../skills/git-workflows/SKILL.md
…

Three variants:

  • ={{skills.all}}= — manifest (default).
  • ={{skills.<id>}}= — full body inline (rare, explicit).
  • ={{skill_path.<id>}}= — just the path string.

Token savings + source-of-truth fidelity + lazy-load matches Claude Code's actual skill loading model.

6.5 Final prompt bundle

At spawn:

  1. System = template's ** Persona + ** Operating Rules.
  2. Skill manifest = result of ={{skills.all}}= injection.
  3. User turn #1 = prompts/<kind>/<state>.org ** Spawn-time injection with slots filled.
  4. Hand to driver.

At state transition mid-run:

  1. Driver session continues (same conversation).
  2. Daemon injects user turn = prompts/<kind>/<new_state>.org ** Continuation injection with slots filled.
  3. Supervisor advances run state; writes run.transitioned tx.

State transition is initiated by a worker-callable tool transition_state(new_state: str, reason: str). Supervisor validates against the per-kind state machine. Invalid transitions return an error result to the worker.

6.6 Project task files (Q9)

Per-project state lives inside the project repo at <project>/.orgasmic/.

<project>/.orgasmic/
  project.org              ← identity + project-global settings
  sprint.org               ← active tasks
  backlog.org              ← unready / future tasks
  archive.org              ← terminal-state tasks (machine-managed)
  tx/YYYY-MM.org           ← project-scoped tx (committed)
  glossary.org             ← project glossary (overrides + extends global, v0.0.5)
  decisions.org            ← project decision graph (v0.0.5)
  architecture.org         ← project architecture nodes (v0.0.5)
  context-map.org          ← per-project bounded-context map (v0.0.6)
  snapshots/<uuidv7>/      ← graph snapshots (uncommitted, v0.0.5)
  risks.org                ← risk register (optional, v0.0.7)
<project>/docs/adr/*.org   ← daemon-generated ADRs (committed, v0.0.6)

A task heading:

* CLAIMED TASK-024 Add retry handling :backend:reliability:
:PROPERTIES:
:ID:                TASK-024
:KIND:              implementer
:STATE:             claimed
:PRIORITY:          P1
:OWNER:             worker-backend-03
:AGENT_TEMPLATE:    backend-implementer-v2
:WRITE_SCOPE:       src/payment_client.py tests/test_payment_client.py
:READ_SCOPE:        src/** tests/**
:DEPENDS_ON:
:BLOCKED_BY:
:TEST_CMD:          pytest tests/test_payment_client.py
:LAST_RUN:          run-abc-123
:LAST_TX:           tx-20260521-001
:LAST_UPDATED:      [2026-05-21 Thu 14:32]
:END:

** Description
** Acceptance Criteria
- [ ] …
** Evidence
[appended by worker]
** Worklog
[appended by worker]
  • TODO keyword = state.
  • Sub-tasks use heading depth (***, ****); inherit parent properties.
  • Workers write only to their own claimed task's ** Evidence and ** Worklog, plus state via transition_state.
  • archive.org machine-managed; daemon moves terminal-state tasks here on configurable schedule (default: immediate).
  • Parse failures don't crash anything; in-memory index keeps last-good.

* BOARD orgasmic projects
:PROPERTIES:

  • No auto-scan; adding is always explicit.

7 Runtime

7.1 Driver layer (Q11a)

Lifted from HAR's src/drivers/. Four drivers in v0.0.1: claude_acp, codex_appserver, hermes, tmux_tui.

Adaptations vs HAR:

  • DriverEvent callbacks map to JSONL session lines via the daemon's per-run writer.
  • Driver registers transition_state(new_state, reason) as a worker tool.
  • Babysitter-kind drivers register poke_implementer, restart_implementer, escalate_to_human, record_finding instead.

Driver restart capability is explicit:

  • tmux_tui runtimes are reattachable when the tmux session still exists.
  • codex_appserver and hermes are reattachable only if their driver reports a stable provider-side session handle.
  • claude_acp is treated as non-reattachable unless ACP exposes a durable resume handle. On daemon restart, such runs become interrupted and are continued by a fresh run, not silently resumed.

7.2 Supervisor (Q11b)

Lifted from HAR. Adaptations:

  • State machine is per-kind, defined in shipped/schema/state-machine.org.
  • Uniqueness invariant: at most one active run per (task_id, kind) tuple (replaces HAR's per-(source_task_id, stage_id)).
  • Lease, timeout, release-with-reason model unchanged.
  • Every live runtime is owned by (run_id, runtime_id, boot_id). Cleanup, timeout, and release paths must update only the runtime instance they own; stale shutdown from a previous daemon boot must not clobber a replacement runtime.

7.3 Event capture (Q11c)

Driver emits DriverEvent on a channel. Per-run writer goroutine consumes the channel and O_APPEND=s JSONL lines to =sessions/<run-id>.jsonl. Writer is the single owner of that file's FD. Controlled daemon restart asks active writers to flush before exit; boot reconciliation treats an unterminated session file as evidence for interrupted, not failed.

7.4 Babysitter pipeline (Q11d, Q7-4)

  • Babysitter runs as a separate process with its own driver + session at sessions/<run-id>.babysitter.jsonl. Shares run-id with implementer.
  • Daemon summarizes the implementer's session stream into chunks and posts them as user-turn messages to the babysitter on a cadence: every N implementer events (default 50) or T seconds (default 60s), whichever first.
  • Babysitter tools:
    • poke_implementer(message) → daemon injects message as a user turn in the implementer's session.
    • restart_implementer(reason) → daemon releases the implementer's lease and respawns a fresh one with the same task.
    • escalate_to_human(reason) → daemon raises a question that surfaces in the UI.
    • record_finding(text) → daemon appends to the implementer's ** Evidence section as a "babysitter note."
  • Babysitters cannot edit code, cannot call orgasmic CLI commands, cannot modify state outside their tool set.

7.5 Spawn paths (Q11e)

All workers spawn through one RPC: acquire_task(task_id, agent_template_id, overrides). Sources:

  • CLI: orgasmic run --task X --template Y [overrides]
  • UI: "Run" button → RPC
  • Manager: orgasmic run via Bash inside its tmux session.
  • Auto: per-project policy in project.org (:AUTO_ACQUIRE: property); daemon's filewatcher detects new tasks in matching states and self-RPCs. (v0.0.8.)

All paths produce the same run.start tx and the same session file.

Continuation after interruption produces run.continue tx with :PREVIOUS_RUN: pointing to the interrupted run. The continuation prompt includes the task heading, the previous session JSONL path, current worktree diff summary, and original acceptance criteria.

7.6 Daemon restart and boot reconciliation

Restart recovery is boot reconciliation, not snapshot restore. The daemon does not roll back graph files on startup and does not auto-resume hidden provider conversations.

Controlled self-hosted restart/update:

  1. Daemon writes daemon.restart_requested tx, pauses new run acquisition, and takes a graph snapshot for manual rollback.
  2. Active session writers flush. Reattachable drivers record their runtime handles; non-reattachable drivers record their last durable session line.
  3. A tiny launcher or service manager starts the rebuilt binary.
  4. New daemon initializes core services, rebuilds the materialized index from tx/org files, then runs boot reconciliation before accepting new run acquisition.
  5. Boot reconciliation scans run records, session files, and provider handles:
    • live reattachable runtime -> run.reattached
    • non-reattachable or missing runtime with non-terminal task -> interrupted
    • terminal task/run pair already recorded -> no-op
    • ambiguous state -> surface a UI question; no destructive action
  6. Manager receives a daemon.recovered tick containing the new boot_id, reattached runs, interrupted runs, dirty-file summary, and pending questions. The manager decides whether to start continuation runs.

Uncontrolled crash follows the same boot reconciliation path, except there is no daemon.restart_requested tx and no guarantee that active writers flushed. The UI should show a soft reconnecting state while the daemon is unavailable; after reconnect it shows the recovered run classifications.

CLI/HTTP mutations that may be retried across restart carry a stable request_id. If a retry finds that the old daemon already committed the tx, the new daemon returns the committed result instead of duplicating the write.

8 Manager (Q12, revised)

The manager is just another agent template with :KIND: manager, runs via the tmux-tui driver (Claude Code or similar TUI agent inside tmux).

8.1 Trigger model (12a)

Event-triggered, with optional cadence. Never always-on. Triggers funnel through one RPC manager.tick(reason, context).

Triggers:

  • File change in a watched .orgasmic/ dir (new task, state change).
  • Worker raises a question.
  • Worker stalled (babysitter escalation).
  • Worker run completes (success or failure).
  • Daemon recovers from restart and reports interrupted or reattached runs.
  • UI button: "Ask the manager."
  • Optional cadence (cron-like, default off).

8.2 Session model (12b)

Each tick spawns a fresh tmux-tui session with persona + relevant skills (manifest+path injection) + current context all injected at the top of the initial payload.

Rationale: tmux-tui drives an external TUI CLI agent whose conversation state lives inside its own memory, not in orgasmic's session log. We cannot reliably resume a tmux-tui session by replaying orgasmic-side events into it.

Continuity across ticks comes from the manager re-reading orgasmic state at the start of each tick (the persona instructs it to do so).

8.3 Permissions (12c)

No permission tiers, no tool whitelist. The tmux-tui agent has access to every MCP and skill the host system has configured.

Discipline lives in the prompt:

  • The persona tells the manager what to do.
  • A manager-tool-discipline skill says "don't use MCPs or skills outside what this prompt references."

No sandbox is built into orgasmic at v0.0.1.

8.4 Tool surface (12d)

The manager has whatever the host system has, including the orgasmic CLI on its PATH. It drives orgasmic through CLI commands. No orgasmic-specific tools beyond the CLI.

8.5 Oversight (12e)

  • The manager is instructed (via persona/skills) to write manager.action tx in listed situations. orgasmic does not hook the manager's actions.
  • No approval queue v0.0.1. Yolo by default. If the manager needs input, it asks in the conversation (regular CLI chat) and waits.
  • Dedicated Manager view in UI: tmux attach to live session + recent manager.action tx tail + pending questions raised to the manager.

8.6 Manager answers worker questions (12f)

From day one. When a worker raises a question (** Question section on its claimed task), the manager — at any tier-equivalent — is triggered with the question context and can call orgasmic question answer <id> "<answer>" from its Bash. Every such action is a manager.action tx + a UI notification "manager answered Q12 on your behalf — review."

13 Design-graph layer (post-v0.0.1, v0.0.5–v0.0.6)

The decision-anchored layer extends orgasmic's substrate from "tasks + runs" to a full design graph: glossary terms → decisions → architecture nodes → tasks → code (via task :PRODUCES: edges + advisory in-code markers), with stale propagation across the graph and a generated-ADR artifact set.

Inspired by:

  • the decision-anchored demo plan at archive/21_05_26_init/decision-anchored-demo-plan-2.md (the 4-layer DAG, reconciliation flip, snapshot safety valve). We deliberately drop that plan's pragma-block / daemon-written-into-source-code mechanic in favor of advisory in-code markers: the graph is authoritative; markers are grep hints. See ** Advisory code markers below.
  • Matt Pocock's grill-with-docs skill (https://github.com/mattpocock/skills/tree/main/skills/engineering/grill-with-docs) for the glossary-first interview methodology, ADR triage gate (hard to reverse + surprising + real trade-off), codebase-exploration substitution, and the principle that CONTEXT is a glossary, not a spec.

This is core orgasmic surface — not an optional plugin — but it ships after v0.0.1 since the existing scope is already a 3–4 month build. v0.0.5 lands the graph + griller + architector + manager-orchestrated reconciliation; v0.0.6 lands the glossary layer + glossarist + ADR triage.

13.1 Two scopes: global and per-project

Same artifacts, two locations. The loader rule (user/ overrides orgasmic/shipped/) extends to a scope rule: project-local artifacts override global ones for term lookups, decision references, and architecture-node naming. The daemon resolves with project-first, global-fallback semantics.

$ORGASMIC_HOME/user/
  glossary.org           ← global vocabulary (used across projects)
  decisions.org          ← global decisions (cross-project reasoning)
  architecture.org       ← global architecture (project boundaries / integrations)
  context-map.org        ← multi-context map for the global scope
  docs/adr/*.org         ← daemon-regenerated global ADRs
$ORGASMIC_HOME/state/
  snapshots/<uuidv7>/    ← global-scope graph snapshots

<project>/.orgasmic/
  glossary.org           ← project glossary (overrides + extends global)
  decisions.org          ← project decision graph
  architecture.org       ← project architecture nodes
  context-map.org        ← multi-bounded-context map for this project
  snapshots/<uuidv7>/    ← project graph snapshots (uncommitted)
<project>/docs/adr/*.org ← daemon-regenerated project ADRs (committed)

A grilling session is scoped at start: global ("what does my dev stack look like?", "where does HAR end and orgasmic begin?", "what's my personal release policy across all projects?") or a specific project (or a bounded context within a project). Cross-scope references are first-class:

  • A project decision may :REFERENCES: a global decision.
  • A project glossary term may :EXTENDS: or :OVERRIDES: a global term.
  • A global architecture node may declare :HOSTED_IN: pointing at a board project (so global architecture knows which project owns which component).

This makes orgasmic usable both as a per-repo design coordinator and as a bird's-eye personal-architecture tool — same UI, same daemon, same agent kinds.

13.2 Four-layer graph + glossary

Layers:

  • Decision — a question with options, exactly one :RECOMMENDED:, optional :CHOSEN:; status pending | decided | stale | superseded.
  • Architecture — a named component with interface + constraints.
  • Task — orgasmic's existing task heading in sprint.org=/=backlog.org. Linked into the graph via :IMPLEMENTS: on the task pointing at architecture or decision IDs.
  • Code — files on disk. Code "nodes" exist in the daemon's index only as derived entries from task :PRODUCES: edges; they have no .org heading of their own. The graph link between design and code is the task layer. Optional advisory in-code markers (see ** Advisory code markers below) provide grep hints during reconciliation but are never authoritative.

Glossary is orthogonal — terms are referenced from any layer via :GLOSSARY_REFS:.

Edges:

  • :DECIDES: decision → architecture | decision (cascade)
  • :COMPOSES: architecture → architecture (parent/child)
  • :IMPLEMENTS: task → architecture | decision
  • :PRODUCES: task → code file path (creates a code-node index entry)
  • :DEPENDS_ON: any layer → any layer (advisory; code-layer must be acyclic)
  • :GLOSSARY_REFS: any → glossary term IDs
  • :REFERENCES: cross-scope link (project → global, or sibling project)

13.3 File formats

All three new files use orgasmic's property-drawer convention. Daemon is the single writer (RPC-only). Not append-only — these are working files with stable IDs that get revised in place.

Decision heading:

* dec_007 Code block highlighting library
:PROPERTIES:
:ID:                 dec_007
:STATUS:             decided
:KIND:               question
:RECOMMENDED:        opt_a
:CHOSEN:             opt_a
:SEMANTIC_HASH:      sha256:abc…
:DECIDES:            arch_005 task_023 task_024
:GLOSSARY_REFS:      hl-lib syntax-highlighting
:ADR_WORTHY:         t
:ADR_NUMBER:         0007
:CODE_REFS:          src/highlight/mod.rs
:CONTRADICTIONS:
:DECIDED_AT:         [2026-06-12 Fri 11:04]
:END:
** Why asked
** opt_a syntect
:PROPERTIES:
:LABEL:              syntect
:RECOMMENDED:        t
:COMPLEXITY:         low
:PREDICTED_OUTCOME:  Mature, large language pack; CSS uses syntect classes.
:IMPLIED_DECISIONS:
:END:
** opt_b tree-sitter
…
** opt_c plain

Architecture heading is analogous, with :INTERFACE: and :CONSTRAINTS: properties + free-form body.

Glossary term heading:

* term:hl-lib Highlighting library
:PROPERTIES:
:ID:                 hl-lib
:CANONICAL:          highlighting library
:AVOID:              syntax engine, lexer lib
:RELATES_TO:         renderer, code-block
:DEFINITION:         A vendored crate that turns source text into spans of
                     CSS class names for HTML rendering.
:OVERRIDES:          global:hl-lib       ← cross-scope (optional)
:DECIDED_AT:         [2026-06-12 Fri 11:00]
:END:

CONTEXT.md-style "avoid these aliases" is first-class: :AVOID: drives the griller's challenge behavior ("you said lexer lib, the glossary says highlighting library — which is it?"). Glossary is a glossary, not a spec — daemon rejects glossary entries whose definition contains implementation references.

context-map.org: pointer list to per-context sub-directories when a scope is multi-bounded-context. Same shape as user/board.org.

13.4 Hashing

Every graph node (decision, architecture, task, glossary) has :SEMANTIC_HASH: computed by the daemon. Code is not hashed — files are not graph nodes.

Layer Semantic input
Decision sha256(questionintent ⨁ chosenoption_id ⨁ implieddecisions)
EXCLUDES prose: whyasked, usernote, option labels.
Architecture sha256(name ⨁ canonicalize(interface) ⨁ canonicalize(constraints))
Task sha256(title ⨁ canonicalize(acceptance))
Glossary sha256(canonical ⨁ avoid ⨁ definition ⨁ relatesto)

A pending decision (:CHOSEN: empty) hashes with chosen_option_id = "".

Editing a node's prose (:DESCRIPTION:, :WHY_ASKED:, :USER_NOTE:, option labels) does not change :SEMANTIC_HASH: and therefore does not cascade-invalidate downstream nodes. This is the "wording-only edit doesn't break the world" rule (formerly SC-5 in the demo plan).

13.5 Stale propagation

Daemon's in-memory index maintains forward + reverse edges. When a heading's content updates:

  1. Compute new :SEMANTIC_HASH:.
  2. If unchanged → done (semantic-input-pinning early-termination).
  3. If changed → BFS over forward edges marking touched headings :STATUS: stale.
  4. Emit graph.stale_propagated event on the WS stream.
  5. Write node.revised and node.staled tx entries.

All atomic per single RPC; partial propagation never observable. Cross-scope edges are walked when a global node is revised.

13.6 Advisory code markers

Code linkage to the design graph is authoritative via task :PRODUCES: edges (graph-side) and advisory via in-code markers (code-side). The daemon never writes inside source files. There is no pragma block, no hash-validation step, no language adapter, no pre-commit hook for code markers.

Marker format — single-line comment, free placement:

<comment-token> orgasmic:<node-id>[:<option-id>][, <node-id>[:<option-id>]]

Examples:

// orgasmic:dec_007:opt_a
pub fn highlight(lang: &str, src: &str) -> Vec<HtmlSpan> { … }
# orgasmic:dec_007:opt_a, dec_010:opt_b
def render_code_block(lang, src): ...
-- orgasmic:dec_021
CREATE INDEX ...

Any layer ID works (dec_*, arch_*, task_*, term:*). The :opt_X is optional. Multi-node on one line is allowed. Placement is the writer's judgment — top of file, above a function, inside a hot block.

Behavioral rules — codified in shipped/conventions/code-markers.org and injected into agent prompts via the slot ={{conventions.code_markers}}=:

  • Implementer agents MAY leave a marker near code that materially expresses a decision they're implementing. It is a courtesy hint for future reconciliation, never required. Don't decorate every function. Don't add markers for code that's only incidentally related.
  • Implementer agents running in reconciling state, when searching for code affected by a changed decision:
    1. Start from :PRODUCES: files on the claimed task — authoritative.
    2. Grep for orgasmic:<decision-id> markers — hints, neither complete nor guaranteed accurate. Files surfaced only by the grep (outside the claimed task's :PRODUCES: set) require an explicit user OK via the ** Question mechanism before editing.
    3. Use general codebase exploration (glossary terms, architecture interface names, symbol search).
    4. Absence of a marker does NOT mean a file is unaffected.
    May update or add markers as a courtesy when editing — never as an obligation.
  • Griller agents in the exploring state also grep for orgasmic:<topic-related-IDs> to see if existing code already commits to a choice the user is being asked about.

The system is honest about its epistemics: it can answer "here are the files we know link to dec007" but never "here are all the files affected by dec007." That's a human/agent judgment call, not a daemon claim.

13.7 ADRs as generated artifacts

ADRs are not a separate authored file format. They are a generated view of decisions where :ADR_WORTHY: t and :ADR_NUMBER: is set.

Output: <scope-root>/docs/adr/<NNNN>-<slug>.org. Daemon regenerates on every decision revision; pre-commit hook orgasmic check adr rejects hand-edits. Sequential numbering is per-scope (global has its own numbering; each project has its own).

Triage rules — griller proposes :ADR_WORTHY: t only when all three hold (verbatim from grill-with-docs/ADR-FORMAT.md):

  1. Hard to reverse — meaningful cost to change later.
  2. Surprising without context — a future reader will ask "why on earth?"
  3. Result of a real trade-off — genuine alternatives existed.

User confirms or overrides in UI. Most decisions in a grilling session do not become ADRs; that's expected.

ADR body (per grill-with-docs/ADR-FORMAT.md): one to three sentences. The value is recording that the decision was made and why. Optional sections (Status, Considered Options, Consequences) only when they add genuine value.

13.8 Snapshots

Cheap file-level copies of the relevant .org files (no Rayforce here — plain copies, atomic via rename).

  • $ORGASMIC_HOME/state/snapshots/<uuidv7>/ for global-scope graph snapshots.
  • <project>/.orgasmic/snapshots/<uuidv7>/ for project-scope graph snapshots.

UUIDv7 = sortable. Auto-snapshot triggers:

  • Before any user-initiated grilling-session start.
  • Before any reconciliation kickoff.
  • Before any controlled self-hosted daemon restart/update.
  • Before any manual restore (so undo-the-undo works).

Manual snapshot via orgasmic snapshot take [--scope X] [--label L]. Restore via orgasmic snapshot restore <id>. Snapshots restore the graph, not files. File edits are git's domain — README documents the recovery path (git restore . after a graph restore, when needed).

13.9 Pipeline: griller → architector → planner → implementer

One agent per layer, each owning its file. The pipeline runs forward during grilling (new design) and runs in reverse on the stale subgraph during reconciliation (design change). Each stage produces a reviewable artifact (a settled .org file or set of edits) before the next stage begins — the user is never confronted with half-baked downstream structure based on premature decisions, and a mid-grilling answer change does not burn tokens on re-proposing architecture that hadn't been proposed yet.

griller    (decisions)
   ↓ grilling.complete
architector (architecture, ADR triage)
   ↓ architecture.complete
planner    (tasks)
   ↓ planning.complete
implementer (code) [N parallel, one per claimed task]

Manager auto-advances each stage when the prior stage completes if the grilling-session setting auto_cascade is true (default); otherwise it pauses at each boundary for user confirmation. The user can pause, edit, or restart at any boundary.

During reconciliation, the manager scopes each agent invocation to the stale subset of its layer — not the whole graph. See ** Reconciliation-session lifecycle below.

13.9.1 griller (decisions only)

Drives grilling sessions. Writes only to decisions.org. Never proposes architecture or tasks. ADR-worthiness is left as a hint that the architector confirms. State machine (shipped/schema/state-machine.org):

exploring → proposing → awaiting_answer → glossary_check
         → contradiction_check → next | complete
  • exploring — before asking, ripgrep + LSP search the relevant codebase (the project's, or all of them if scope is global) to see if the answer is already determined. If yes: write a "found in code" decision with :CODE_REFS: populated, :STATUS: decided, skip the user. (Verbatim from grill-with-docs/SKILL.md: "If a question can be answered by exploring the codebase, explore the codebase instead.")
  • proposing — write the next batch (size = questions_per_round) or single question (mode: socratic) via /api/nodes. May set :ADR_WORTHY: proposed as a hint — the architector confirms or overrides during its own pass.
  • awaiting_answer — poll-free; filewatcher signals when the user answers in UI.
  • glossary_check — undefined term appeared in the answer? Propose canonical + avoid + definition; write atomically with the decision update. Conflict against existing glossary? Surface verbatim: "Your glossary defines X as A, but you seem to mean B — which is it?"
  • contradiction_check — answer disagrees with glossary or code? Populate :CONTRADICTIONS: and surface in the next turn.
  • complete — completeness predicate is simply: every decision in scope has :STATUS: decided and no :CONTRADICTIONS: remain unresolved. No architecture or task checks. Calls POST /api/grilling/complete.

13.9.2 architector (architecture + ADR triage)

Auto-spawned by the manager on grilling.complete for the same scope (or on a stale-arch event during reconciliation). Reads all decided decisions, drafts architecture nodes once, presents the proposal to the user, revises on feedback, completes. Also owns ADR triage. Writes only to architecture.org and to the :ADR_WORTHY: / :ADR_NUMBER: properties of decision headings (the one cross-file write it is permitted).

State machine:

reading → drafting → adr_triage → presenting → revising? → complete
  • reading — load all decided decisions in scope + any existing architecture nodes.
  • drafting — for each decision (or cluster), propose architecture nodes via POST /api/nodes:
    • Set :INTERFACE: based on the chosen option's mechanical implications.
    • Set :CONSTRAINTS: from option metadata + glossary-term semantics.
    • Set :DECIDES: back-edges from each motivating decision.
    • Propose :COMPOSES: parent/child relationships where natural.
    Components implied by multiple decisions are consolidated. The architector sees the design holistically rather than per-decision, which is the whole reason it runs after grilling and not during.
  • adr_triage — apply the grill-with-docs three-test (hard to reverse + surprising without context + result of a real trade-off) to every decision in scope. Override the griller's :ADR_WORTHY: proposed hints with t or nil as appropriate. Assign sequential :ADR_NUMBER: within the scope (global has its own counter; each project has its own). Triggers POST /api/adr/regenerate so docs/adr/*.org reflects the current set.
  • presenting — surface the proposed structure via the ** Question mechanism on a synthetic architect-<scope> task: /"I propose N architecture nodes: arch001 Parser, arch002 Renderer, … . I marked M decisions as ADR-worthy: dec007, dec010, … . Reply `accept`, `revise`, or describe changes."/
  • revising — user feedback → rework. Each revision pass is a fresh drafting → adr_triage → presenting cycle scoped to the user's objections.
  • complete — calls POST /api/architecture/complete.

Token-efficient by design: runs once per scope after grilling completes. Mid-grilling decision answers do not spawn the architector; only grilling.complete does. During reconciliation, the architector is re-spawned with the stale arch subset, not the full graph.

13.9.3 planner (tasks; existing v0.0.1 kind, new responsibility)

planner already exists in the v0.0.1 KIND taxonomy. In the design-graph layer it gains a specific job: read decided decisions + decided architecture in scope, propose tasks with :IMPLEMENTS: edges back into architecture and :PRODUCES: file paths. Writes to sprint.org and backlog.org. Auto-spawned by manager on architecture.complete.

State machine mirrors the architector's:

reading → drafting → presenting → revising? → complete

Calls POST /api/planning/complete on completion. During reconciliation, re-spawned with the stale task subset.

13.9.4 implementer (code; existing v0.0.1 kind, unchanged)

Reads its claimed task, edits files, may leave advisory markers per shipped/conventions/code-markers.org. Auto-spawned by manager once tasks become claimed (existing v0.0.1 mechanism). N implementers may run in parallel, one per claimed task, bounded by project :MAX_CONCURRENT_RUNS:. No changes from v0.0.1 semantics.

13.9.5 glossarist (glossary maintenance)

Background glossary maintenance — manager-triggered tick, not a long-running worker. Reads all .org artifacts in scope, flags:

  • Terms used in headings/bodies but not defined.
  • Glossary entries with zero references.
  • Aliases in :AVOID: lists that someone used as a heading.
  • Cross-scope term drift (project canonical conflicts with global).

Proposals only — never writes glossary terms autonomously. Surfaces a ** Question on a synthetic meta-glossary task.

(Renamed from lexicographer. Glossarist matches the narrowness of the job — tending the glossary, not the broader work of cataloging the language of the system.)

13.10 Grilling-session lifecycle

  1. User clicks Start grilling in UI (scope picker: global / project / bounded-context-within-project).
  2. Modal: topic, questions_per_round (1–5, default 3), outcome_depth (1–3, default 1), mode (socratic | batch), auto_explore_code (t | nil), glossary_strict (t | nil), auto_cascade (t | nil, default t). Socratic mode forces questions_per_round=1, outcome_depth=1. auto_cascade=t means: on grilling.complete, manager auto-spawns architector; on architecture.complete, auto-spawns planner. With auto_cascade=nil, manager pauses at each stage boundary and waits for user confirmation.
  3. UI calls /api/snapshots → daemon takes auto-snapshot.
  4. Manager spawns a griller worker (typically claude-acp driver) with the chosen scope + settings injected into the spawn-time prompt.
  5. Griller loops: explore → propose → user answers in UI → next.
  6. User may revise already-decided decisions live; if downstream architecture or tasks have already been proposed, the daemon marks them stale and the manager kicks off the reconciliation flow (** Reconciliation-session lifecycle below). If only decisions exist so far, no cascade — the architector has not yet run.
  7. When the griller's completeness predicate is met (every decision :STATUS: decided, no unresolved :CONTRADICTIONS:), it calls POST /api/grilling/complete. Per auto_cascade, manager either spawns the architector immediately or pauses for user confirmation.

Sync mechanism — orgasmic's manager loop replaces the demo plan's "clipboard-prompt + user types done" handoff. Filewatcher signals when the user answers; the griller is unblocked by the manager tick. No clipboard, no done-signal. (See * Manager for the underlying tick model.)

13.11 Reconciliation-session lifecycle

Reconciliation is not a single agent run — it's the manager re-running the pipeline scoped to the stale subgraph. There is no reconciler kind.

  1. User revises a decided decision in UI (clicks different option + confirm), or revises any node via the .org editor.
  2. UI calls /api/snapshots → auto-snapshot.
  3. UI calls /api/revisions/<id> → daemon flips status, recomputes :SEMANTIC_HASH:, propagates :STATUS: stale over forward edges.
  4. Manager sees the stale set on the WS stream. For each layer with stale nodes, in topological order (architecture before tasks before code): a. Stale architecture nodes → spawn architector in reconciling state (prompt template architector/reconciling.org). Architector reads stale arch + their motivating decisions, either revises each (POST /api/revisions/<id>) or validates unchanged (POST /api/validate/<id>) if :INTERFACE: + :CONSTRAINTS: survive the decision change. Re-runs adr_triage over the affected decisions. b. Stale tasks → spawn planner in reconciling state. Planner revises task bodies, :PRODUCES:, and :ACCEPTANCE: as needed; may create new tasks or supersede old ones. c. Stale code (via stale tasks' :PRODUCES: plus advisory-marker grep hits across the codebase) → spawn N implementer runs, one per affected task. Each implementer reads its claim, edits files, optionally updates advisory markers, transitions to verifying, then done. Implementers may also touch files surfaced only by advisory-marker grep, but must explicitly raise a question to the user before doing so (those edits are outside the task's :PRODUCES: set).
  5. Each layer's agent presents a plan via ** Question before destructive edits; user must reply before the agent advances. Layers run strictly in order; a layer doesn't start until the prior layer reports completion (architecture.complete, planning.complete, all implementer runs done).
  6. When the stale set is empty, manager calls POST /api/reconcile/complete → daemon validates graph-side gates (decision pending-state, trace consistency, file-deletion ⇔ supersede pairing, cross-scope reference validity) and flips stale → decided (or returns violations to the manager, which surfaces them as a question to the user).
  7. User (or manager, per project setting :AUTO_GIT_COMMIT:) commits.

The daemon never invokes git. Git is the operator's responsibility.

The daemon never invokes git. Git is the operator's responsibility (same invariant as the demo plan).

/api/reconcile/complete validation gates (graph-side only):

  • No decision node referenced in node_ids is pending.
  • No decision node anywhere in the touched subgraph is pending.
  • For every superseded code-file reference: the file does not exist on disk. For every non-superseded code-file reference: the file exists.
  • Trace consistency: every :PRODUCES: file path is reachable; no cross-scope :REFERENCES: points at a deleted/superseded node.

Note: there is no physical-hash check on edited files. The reconciliation agents (architector / planner / implementer in reconciling state) are trusted to have done the right edits; user review at each agent's presenting or planning state is the human gate.

13.14 Critical invariants (extends * Defaults)

  1. Daemon is the sole writer of glossary.org, decisions.org, architecture.org, and docs/adr/*.org. Workers and humans go through RPC.
  2. The daemon never writes inside source files. In-code markers are written by agents as a convention, not by the daemon as a guarantee.
  3. apply_revision is atomic. Hash + propagation + tx write all-or-nothing.
  4. Finalize endpoints reject: pending decisions in the touched subgraph, trace inconsistency, file-deletion ⇔ supersede mismatches, cross-scope references to deleted/superseded nodes. (No code-side hash check.)
  5. Nodes are never deleted. Use :STATUS: superseded. Append-only at the node level.
  6. A decision's semantic hash never depends on prose. Editing prose-only doesn't cascade.
  7. Code is not a graph node. Files are linked via task :PRODUCES: edges and (advisorily) by in-code markers. There is no code-layer hash.
  8. The agent never marks a pending decision decided. Only the user clicks.
  9. An auto-snapshot precedes every user-initiated grilling or reconciliation kickoff.
  10. Cross-scope references are advisory at write-time but enforced at finalize-time. Project decisions referencing a deleted global decision block reconcile-complete.
  11. Glossary entries contain no implementation details. Daemon rejects glossary writes that include code paths, type signatures, or function names in :DEFINITION: — CONTEXT is a glossary, per grill-with-docs.
  12. Code markers are advisory, never authoritative. Reconciliation agents must not treat the absence of a marker as proof a file is unaffected.

15 Open questions for v0.0.1 implementation

Items the design intentionally did not pin, to be decided during build:

  • Exact wire format for the bearer-token endpoint scheme (query param vs header for WS — HAR's pattern probably works).
  • Whether orgasmic init writes config.yaml in TOML or YAML (current default is YAML to match hermes; revisit if YAML parser disagreement bites).
  • Whether to use notify or notify-debouncer-full for the watcher.
  • Concrete CodeMirror org-mode highlighter — there are several; pick one and fork the highlights file.
  • Concrete shape of the LAN-bind warning (should require explicit confirm).
  • Whether the daemon-state directory ever moves to SQLite for indexing performance (post-v0.1 if .org parsing dominates the watcher CPU).

Design-graph layer (v0.0.5–v0.0.6) — to pin during that slab:

  • Cross-scope snapshot semantics: should a global-scope reconciliation auto-snapshot every affected project's graph too, or only the global one?
  • Whether socratic mode should be the default for global grilling (stakes high) and batch the default for per-project (bandwidth matters), or whether the modal should require an explicit pick each time.
  • Whether :CHOSEN: multiple is allowed (a decision picking multiple compatible options, e.g., "support both X and Y") or whether that's two separate decisions.
  • How glossarist ticks should be triggered: filewatcher debounce vs manager-scheduled cron-style (default off) vs explicit CLI only.
  • Whether docs/adr/ should also be exported as Markdown (.md) for GitHub-renderer compatibility, in addition to .org.
  • Whether implementer agents in reconciling state should surface files they found via marker grep but couldn't reach via the claimed task's :PRODUCES: as a special "advisory hits" list for the user to confirm in one batch — or raise a separate ** Question per file.
  • Whether grilling sessions can be paused and resumed across orgasmic daemon restarts, or whether a stalled session always starts fresh from the auto-snapshot.
