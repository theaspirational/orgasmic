* Convention: contributing changes

How to make and record changes in this repo so the durable state stays the
source of truth.

- Follow the constraints in `project.org`.
- Keep personal data out of durable state: `tx/`, `decisions.org`, and task evidence are committed and may be published. Use a stable handle (e.g. a GitHub username), not a personal email, in `:ACTOR:`/`:SET_BY:` fields; write paths relative to the repo or as `$ORGASMIC_HOME`/`~`, never absolute home paths; don't record machine hostnames or other environment identifiers. Machine-specific notes belong in the gitignored `tmp/local_instructions.org`.
- Record durable rationale in `decisions.org`, never as stray markdown docs.
- While the project is greenfield (pre-stable), treat the decision and architecture graph as a *living model*: edit nodes in place to keep them true. Use `orgasmic decision create` / `orgasmic architecture create` for genuinely new rationale, and `orgasmic node body set` / `orgasmic node prop set` to revise, merge, correct, or re-link existing nodes. Reserve append-only supersession (a new record plus a `:SUPERSEDES:` backref) for after the graph stabilizes, when history must be preserved. Use only the canonical edge keys (`:MOTIVATED_BY: :RELATES_TO: :DEPENDS_ON: :BLOCKED_BY: :IMPLEMENTS: :PRODUCES: :GLOSSARY_REFS:`); don't invent new ones.
- If something about the project surprises you, add an entry to `gotchas.org`.
- When you change a task's scope, keep `tasks/` current.
