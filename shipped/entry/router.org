* Entry

** Agent router
Worker route when the prompt is a strict orgasmic worker dispatch with all of:
- worker role: implementer, architect, planner, reviewer, or griller
- task ID or task heading
- read/write scope
- acceptance, reporting, or evidence expectations

Worker startup:
1. Read the task heading.
2. Read `project.org` and `gotchas.org`.
3. Read only referenced conventions and source files.

Manager route for every other prompt, including arbitrary requests,
pickup/status requests, and prompts beginning with `/orgasmic`.

Manager startup:
1. Check `~/.agents/skills/orgasmic/SKILL.md` or the visible skill registry.
2. If installed, read the skill and follow the selected `/orgasmic` subcommand.
3. If absent, offer `/orgasmic install`; keep `.orgasmic/` read-only until the
   runtime is available.

Manager-implementer: when the manager edits source directly in this session
instead of dispatching a worker (single-worker setups), read
`conventions/manager-implementer.org` before the first source edit.

** Project map
- `project.org`: baseline, mission, operating constraints.
- `tasks/`: `backlog.org`, `todo.org`, `in_progress.org`, `in_review.org`, `done.org`, `cancelled.org`, `goal.org`, `handoff.org`.
- `tasks/goal.org`: manager focus only.
- `tasks/handoff.org`: manager continuity only.
- `decisions.org`: durable rationale.
- `architecture.org`: architecture graph, source paths, node tests.
- `glossary.org`: domain language.
- `gotchas.org`: repeated traps; read before source edits.
- `conventions/`: repo-local working agreements.
- `tx/`: agent activity log.

** Write rules
- Before `.orgasmic/` state writes: run
  `command -v orgasmic >/dev/null 2>&1 && orgasmic status >/dev/null 2>&1`.
- If it succeeds: write through `orgasmic ...` CLI commands.
- If it fails: stop - install or start the runtime (`/orgasmic install`,
  `orgasmic daemon start`); do not hand-edit.
- Add repo-specific conventions as separate files under `conventions/`.

** Contributing discipline
- Follow the constraints in `project.org`.
- Keep personal data out of durable state: `tx/`, decisions, task evidence, and
  worklogs may be committed or published. Use stable handles instead of personal
  email, prefer repo-relative paths, and keep machine-specific notes in
  gitignored `tmp/local_instructions.org`.
- Record durable rationale in `decisions.org`; do not create stray decision
  docs.
- While the project is greenfield, treat decisions and architecture as a living
  model: create genuinely new rationale through the CLI, and revise existing
  nodes through `orgasmic node body set|append` or `orgasmic node prop set`.
- If something about the project surprises you, add or ask for a `gotchas.org`
  entry. When task scope changes, keep `tasks/` current through the CLI.

** Tooling usage
- Use `orgasmic ...` as the agent and human surface; treat the daemon as the CLI
  backend and write authority for `tx/` and sessions.
- `/orgasmic recall` means brief and stop.
- `/orgasmic resume` means brief, then continue the next clear action.
- Use `orgasmic --help` and the live worker list instead of stale local copies
  when choosing commands or worker ids.

** Dispatch rule
- Implement and review work through orgasmic workers, never by shelling out to a
  raw provider CLI such as `codex`, `claude`, or `cursor-agent` for project
  work. Raw provider CLIs bypass daemon write authority, tx recording, worktree
  isolation, dispatch leases, and reattach/recovery.
- A request naming provider, model, or effort maps to
  `orgasmic manager dispatch --kind <kind> --worker <worker-id> --model <model>
  --effort <effort> --brief <path>`, resolving `<worker-id>` from
  `orgasmic worker list`.
