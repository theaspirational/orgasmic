** Description
Umbrella task that turns a freshly scaffolded `.orgasmic/` into real project
state by auditing this repository and grilling the operator. It owns no file of
its own; it completes when its four subtasks are DONE.

Run the subtasks in order — TASK-C9V29.1 -> TASK-C9V29.2 -> TASK-C9V29.3 ->
TASK-C9V29.4. Each one:
1. Audits the codebase for concrete evidence (READMEs, manifests, source layout,
   tests, config) instead of guessing.
2. Grills the operator in small batches of high-value questions to confirm
   intent and fill the gaps code cannot answer.
3. Writes its target `.orgasmic/` file with real records — not placeholders — so
   later agents have a worked example to follow.
4. Leaves an activity record via `orgasmic tx record`.

Requires a running daemon and CLI (`/orgasmic install`, then `orgasmic daemon
start`). Bootstrap `.orgasmic/` writes use `orgasmic node body set|append`,
`orgasmic decision create`, `orgasmic architecture create`, and `orgasmic task
update` — see `orgasmic --help`. Replace this task with your own once bootstrap
is complete.

- [ ] TASK-C9V29.1, TASK-C9V29.2, TASK-C9V29.3, and TASK-C9V29.4 are all DONE.
- [ ] `project.org`, `decisions.org`, and `architecture.org` hold real,
      repo-specific records beyond the scaffolded `dec_K1DR7` bootstrap decision
      (no `[Replace ...]` placeholders remain).
- [ ] `.orgasmic/tx/` has at least one activity record from this bootstrap.

** Description
Establish the project baseline. Audit the repo for what it is and does, then
grill the operator on the intent the code cannot reveal.

Audit: top-level READMEs and docs, build manifests and language(s), directory
layout, entry points, test/lint/build commands, and any existing product or
design notes.

Grill: the mission in 2–4 sentences; the operating constraints every worker must
respect; primary users and surfaces; and — only if this repo ships a user-facing
product — the product baseline and design register. Ask in small batches; keep
known facts, inferred assumptions, and open questions separate.

Write: fill `.orgasmic/project.org`, replacing the `** Mission` and
`** Operating Constraints` placeholders with confirmed prose. Capture
repo-specific verification and branch notes there or in conventions as prose.

** Acceptance Criteria
- [ ] `** Mission` states the real goal in 2–4 sentences.
- [ ] `** Operating Constraints` lists concrete, repo-specific constraints.
- [ ] Every claim is grounded in code evidence or an explicit operator answer;
      no invented requirements.
- [ ] No `[Describe ...]` / `[Project-specific ...]` placeholders remain.

** Description
Recover the consequential, non-obvious decisions already embedded in this
codebase and confirm their rationale with the operator.

Audit: look for choices a newcomer would otherwise re-litigate — storage and
file substrate, module boundaries, sync vs async seams, dependency and framework
picks, error-handling and concurrency stance, build and release shape.
Distinguish settled decisions from still-open questions.

Grill: for each candidate, confirm Context (what forced the choice), Decision
(what was chosen), and Consequences (what it commits the project to). Drop
anything the operator calls incidental rather than a real decision.

Write: create ADR-style records with `orgasmic decision create` — the daemon
mints each `dec_XXXXX` id; capture the returned id before writing any prose
that references it (create decisions first, reference them after; invented
sequential ids like `dec_001` are rejected on create). Each record carries
`:ID:`, `:DECIDED_AT:`, topic tags, and `** Context` / `** Decision` /
`** Consequences`. Presence means accepted — there is no status field. Seed any
new domain terms you rely on into `.orgasmic/glossary.org`.

** Acceptance Criteria
- [ ] At least 3 repo-specific decision records beyond `dec_K1DR7`, each
      grounded in code or an operator answer (no speculative decisions).
- [ ] Each record has Context / Decision / Consequences plus topic tags.
- [ ] Domain terms referenced by the decisions exist in `glossary.org`.

** Description
Move tacit knowledge out of pre-existing instruction files so every rule has
exactly one canonical `.orgasmic/` home. If the repo has no instruction files
beyond the scaffolded root pointers, record a no-op and close.

Audit: pre-existing agent instructions (`CLAUDE.md`, `AGENTS.md`,
`.cursorrules`, `.github/copilot-instructions.md`) and contributor docs or
specs (`CONTRIBUTING.md`, `docs/`) for rules an agent is expected to obey.

Grill: for each rule, confirm with the operator that it is still live, then
route it to its home: working agreements -> `conventions/`; repeated traps ->
`gotchas.org`; choices with rationale -> `decisions.org`; mission or operating
constraints -> `project.org`; machine-specific setup ->
`tmp/local_instructions.org` (local, ignored). Drop rules the operator calls
dead.

Write: with operator approval, shrink each migrated instruction file down to
the `.orgasmic/entry.org` pointer plus whatever the operator keeps for human
readers. A rule must not survive in two homes.

** Acceptance Criteria
- [ ] Every live rule from pre-existing instruction files has exactly one
      `.orgasmic/` home — or the Worklog records a no-op because no such
      files existed.
- [ ] Machine-specific notes live in `tmp/local_instructions.org`, not in
      committed files.
- [ ] Migrated instruction files route to `.orgasmic/entry.org` and no longer
      restate migrated rules.
