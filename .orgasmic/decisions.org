* dec_K1DR7 Product boundary    :product-scope:
:PROPERTIES:

** Context
The first design question is whether orgasmic is a standalone product, a HAR migration, or a hosted collaboration system. The answer shapes whether product concerns mix with generic agent coordination and whether identity and hosting requirements dominate the first version.
** Decision
Orgasmic is a standalone local developer coordination app. It owns its own app model while borrowing selected HAR execution patterns and Org-style file primitives. It is not a HAR migration, not an Emacs workflow, and not a hosted team SaaS.

* dec_XSV21 Home and install layout    :install-distribution:
:PROPERTIES:
:ID:            dec_XSV21
:GLOSSARY_REFS: term_CJ2R4 term_WQMJJ term_DVSS2 term_TDNG3
:DECIDED_AT:    [2026-05-21 Thu 19:06]
:SOURCE:        archive/21_05_26_init/spec.org::*Install and data model
:END:
** Context
The install shape determines update safety, user customization, and how orgasmic can manage itself. A repo-root-as-home layout collides updates with user data, while package-manager-first weakens local source hackability during pre-v0.0.1 work.
** Decision
Use a two-layer local boundary. The installed app provides shipped code and default content; =$ORGASMIC_HOME= holds user data, overrides, runtime state, sessions, secrets, logs, config, and the CLI entrypoint. In the dogfood init, =~/.orgasmic/orgasmic= points at this checkout. The loader always prefers user overrides over shipped defaults.
** Consequences
Install and update code must preserve the home directory while refreshing app binaries, shipped content, and source checkouts. The day-one user package is macOS-first: a prebuilt Tauri desktop distribution bundling workbench, local daemon sidecar, CLI sidecar, and shipped content; the source checkout under =$ORGASMIC_HOME/orgasmic= becomes contributor-mode fallback (README leads with the desktop app, fork/source setup moves to CONTRIBUTING). App and daemon are separate runtime roles: for Mac-local projects the app supervises a bundled local daemon, while for remote Linux projects the daemon is installed and supervised on that server (owning the server-side =$ORGASMIC_HOME=, managing remote repos, running workers there) with the macOS app connecting as a client profile. Without a paid Apple Developer Program account the v0.0.1 track is an Apple-Silicon-only, ad-hoc-signed tester DMG with documented Gatekeeper approval; the primary install path is a terminal installer that downloads the DMG with =curl=, copies =orgasmic.app= into an Applications directory (defaulting to =~/Applications=), verifies the ad-hoc signature, and opens the app, normally avoiding browser-applied =com.apple.quarantine= metadata. The desktop app uses Tauri updater artifacts signed with an off-repo key; an update installs only after daemon recovery status confirms no live runs are in flight. Update channels are modeled as GitHub release tags (=stable= and =nightly= via =/releases/download/<channel>/latest.json=), keeping the Tauri updater target as the platform key.

* dec_WH9PD Write authority and concurrency    :file-substrate:
:PROPERTIES:
:ID:            dec_WH9PD
:GLOSSARY_REFS: term_N5SJT term_YC32J term_P94BF term_H5YHC
:DECIDED_AT:    [2026-05-21 Thu 19:06]
:SOURCE:        archive/21_05_26_init/spec.org::*Write surface
:END:
** Context
The system needs a clear answer to who may mutate each file role. Append-only everywhere gives strong audit semantics but poor ergonomics for tasks, prompts, and templates; direct edits everywhere give simple files but weak auditability and race protection for graph state.
** Decision
Use role-based write authority. The daemon serializes tx, sessions, graph files, snapshots, and generated artifacts, while selected working files remain directly editable through RPC and advisory flock. Workers mutate only their own claimed task's evidence and worklog directly, with state transitions mediated by the daemon.
** Consequences
Graph and audit integrity have one writer, but task bodies and user-authored content stay ergonomic. Daemon-owned artifacts are RPC-only; other working files use advisory flock.

* dec_DK7AZ Manager runtime model    :manager:
:PROPERTIES:
:ID:            dec_DK7AZ
:GLOSSARY_REFS: term_X4JGJ term_KH8TX term_FM60Y term_465BG
:DECIDED_AT:    [2026-05-21 Thu 19:06]
:SOURCE:        archive/21_05_26_init/spec.org::*Manager (Q12, revised)
:END:
** Context
The manager needs to coordinate state changes without becoming a hidden always-on brain. An always-on resident manager gives conversational continuity but harder recovery and hidden state drift; a daemon-embedded manager mixes orchestration policy with daemon state integrity.
** Decision
The manager is a fresh agent launched on demand through a driver (tmux / acp / …) and the orgasmic CLI — not an always-on brain, and not ticked on a schedule. It bootstraps by running the system-wide =/orgasmic resume= skill, which rereads on-disk =.orgasmic/= state plus the manager doctrine (=manager.org= + the manager conventions); continuity comes from the files and index, never hidden model state. The daemon spawns/opens the session but injects no compiled per-tick prompt (see dec_WTM5Y).
** Consequences
Manager behavior stays outside daemon state integrity. The Manager UI attaches to live sessions and shows recent manager actions; hidden long-lived model state is never treated as source of truth. Because the manager reconstructs its own context via the skill, the daemon needs only to acquire/open a session — no manager prompt compilation lives in the daemon.

* dec_X6Q0S Design graph model    :design-graph:
:PROPERTIES:
:ID:            dec_X6Q0S
:GLOSSARY_REFS: term_P94BF term_H5YHC term_RG690 term_HQJWT
:DECIDED_AT:    [2026-05-21 Thu 19:06]
:SOURCE:        archive/21_05_26_init/spec.org::*Design-graph layer
:END:
** Context
The post-v0.0.1 graph must represent decisions, architecture, work, and code links without turning source code into daemon-owned state. A decisions-and-tasks-only model loses the design bridge between tradeoffs and implementation, while a full code graph pulls orgasmic into physical code ownership and hash validation.
** Decision
The design graph is decisions → architecture → tasks → code, with glossary terms orthogonal to every layer. Code is not a graph node; code linkage is through task =:PRODUCES:= edges and optional advisory markers. Architecture nodes admit a sub-layer of functional-component leaves named =arch_NNN.M= under each =arch_NNN= subsystem, each carrying =:SOURCE_PATHS:= (1–10 files) and typed edges (=:READS:=, =:WRITES:=, =:EXPOSES_REST:=, =:EXPOSES_WS:=, =:SUBSCRIBES_TO:=, =:SPAWNS:=, =:CALLS:=, with =:DEPENDS_ON:= as a discouraged escape hatch); code remains the target of leaf back-edges, never an authored node.
** Consequences
Orgasmic can reason over design impact without claiming ownership of source files; reconciliation starts from graph edges and uses advisory markers only as hints. The design-graph's auto-consistency machinery (semantic-hash, stale propagation, finalize-validate gate, generated ADRs, snapshots) was later removed: the architecture graph survives as an authored, materialized, React-Flow-visualized graph; the glossary as a plain authored doc; and decisions as ADR records.

* dec_TEKEB Grilling-to-architecture pipeline    :design-graph:
:PROPERTIES:
:ID:            dec_TEKEB
:GLOSSARY_REFS: term_X4JGJ term_P94BF term_H5YHC term_RG690
:DECIDED_AT:    [2026-05-21 Thu 19:06]
:SOURCE:        archive/21_05_26_init/spec.org::*Pipeline: griller -> architector -> planner -> implementer
:END:
** Context
Orgasmic needs a durable handoff from user answers to architecture and implementation. A single universal design agent reduces orchestration but blurs file ownership and produces premature downstream proposals; manual human conversion gives clear control but no autonomous architecture pipeline.
** Decision
Use one agent per graph layer: griller owns decisions, architector owns architecture and ADR triage, planner owns tasks, implementers own code. The manager advances a stage only after the prior layer produces a reviewable artifact. The current state represents a completed griller pass.
** Consequences
File ownership stays clear and token use stays bounded; mid-grilling answer changes do not waste work on downstream architecture that should not exist yet. Architector composes architecture via mechanism-split decomposition — each subsystem yields 2–6 functional-component leaves whose nodes are stable contracts (a trait, a REST endpoint group, a watcher loop, a hook, a parser) rather than features or pages — guided by ten encoded principles (stable contract per leaf; verb-namable; mechanism not feature; 1–10 source paths; 2–6 children per parent; no mechanism-as-mechanism or cross-cutting-concern leaves; tests/build scripts/fixtures excluded from =:SOURCE_PATHS:=; bootstrap from existing module boundaries; target/planned nodes kept visually separate and excluded from drift enforcement). The design-graph's auto-consistency machinery (semantic-hash, stale propagation, finalize-validate gate, generated ADRs, snapshots) was later removed; the architecture graph survives as an authored, materialized, React-Flow-visualized graph, the glossary as a plain authored doc, and decisions as ADR records.

* dec_BP4NH Code linkage policy    :code-markers:
:PROPERTIES:
:ID:            dec_BP4NH
:GLOSSARY_REFS: term_HQJWT term_RG690 term_N5SJT
:DECIDED_AT:    [2026-05-21 Thu 19:06]
:SOURCE:        archive/21_05_26_init/spec.org::*Advisory code markers
:END:
** Context
Orgasmic needs traceability to source code without mutating source files behind the developer's back. Daemon-managed pragmas give stronger traceability but unsafe source mutation and a language-adapter burden; no code markers gives cleaner files but weaker reconciliation discovery.
** Decision
Task =:PRODUCES:= edges are the authoritative design-to-code link; in-code markers are advisory. The daemon never writes source markers and never treats a missing marker as proof a file is unaffected. Files surfaced only by marker grep require explicit user approval before an implementer edits them.
** Consequences
Agents may leave useful grep hints, but source ownership stays with implementers and humans. Inline markers gain a second role for architecture-component leaves: the file-level marker =// arch: arch_NNN.M= (or =// @arch arch_NNN.M= in TypeScript) at file top is the authoritative back-edge between code and the architecture model, paired with =:SOURCE_PATHS:= on the leaf. A drift comparator enforces both directions on implementation-backed leaves — every path in =:SOURCE_PATHS:= must exist and carry the matching marker, and every marker found must be enumerated in its leaf. Files may carry multiple markers as a comma-list; granularity is file-level in v1 (item-level deferred). The advisory write-scope role is retained for marker-only files outside =:PRODUCES:= sets.

* dec_CSKBD Self-hosted daemon restart recovery    :run-recovery:
:PROPERTIES:
:ID:            dec_CSKBD
:GLOSSARY_REFS: term_Z672G term_N400W term_A8GT3 term_X4JGJ term_KH8TX term_N5SJT
:DECIDED_AT:    [2026-05-21 Thu 19:26]
:SOURCE:        archive/21_05_26_init/spec.org::*Daemon restart and boot reconciliation
:END:
** Context
Self-hosted orgasmic development can rebuild or restart the daemon while the manager and workers are coordinating changes to orgasmic itself. Daemon-owned auto-resume would require understanding each provider's hidden conversation state and could duplicate edits; failing all workers on restart is simple but brittle; snapshot-restore-by-default is destructive and confuses recovery with explicit rollback.
** Decision
Restart recovery is non-destructive boot reconciliation, not snapshot restore or hidden provider auto-resume. The new daemon rebuilds truth from disk, reattaches only provably live runtimes, marks non-reattachable ACP-style workers interrupted, and asks the manager to continue them from task state, transcript, and git diff.
** Consequences
Self-hosted updates can feel like a reconnect without destructive rollback. The daemon stays conservative; the manager owns continuation. ACP-style workers that die during restart continue through fresh runs seeded with durable task state, prior session JSONL, current diff, and acceptance criteria.

* dec_2D5BC Daemon must provision tool PATH for harness and driver binaries, not inherit the service manager's minimal PATH
:PROPERTIES:
:ID: dec_2D5BC
:TAGS: daemon,install,dispatch,launchd,reliability
:END:

Context: The daemon's generated service definitions set EnvironmentVariables to only ORGASMIC_HOME (crates/orgasmic-cli/src/daemon_service.rs: render_macos_launch_agent plus the systemd-unit and windows-task siblings). select_adapter_for_host keys on host platform only, so on macOS every install - regular bundle and contributor source alike - is managed by a launchd LaunchAgent. launchd then runs the daemon with PATH=/usr/bin:/bin:/usr/sbin:/sbin, so it cannot find the worker driver (tmux/rmux) or the harness (claude), and every manager dispatch comes up inert (inert_reason tmux_missing, then binary_missing:claude).
