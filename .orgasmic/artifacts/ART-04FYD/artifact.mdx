<Section title="Why this artifact exists">

<RichText>
On 2026-07-06 an external grilling session analyzed an **orgasmic snapshot at commit `ad38043`** and produced a report proposing a new product direction: turn orgasmic from a *per-developer local coordination app* into a **local-first lifecycle cockpit**. The report survives at `archive/grilling-session-local-first-cockpit-report.md`; the work products it describes (17 ADRs, a `CONTEXT.md` glossary, a design-fixture Program workspace, and a `program.rs` parser/validator slice with passing tests) live in a **separate workspace that is not on this machine** and are not in this repo.

Meanwhile `main` has moved **117 commits past** the analyzed snapshot — mostly in the opposite stylistic direction: cutting surfaces (the Graph page was excised under `dec_ZG9B9`) and hardening the dispatch/artifact pipeline.

This brief presents the concept **neutrally**, lays out the tensions with our recorded decisions, and ends with the questions the team needs to answer. Nothing here is adopted; no decision has been recorded.
</RichText>

</Section>

<Section title="The concept in one page">

<RichText>
The report's claim: orgasmic's current model is a good **agent workbench** but cannot express *lifecycle* — a cross-repo effort moving through discovery → planning → implementation → review → release readiness, with risks, architecture deltas, and **proof** attached. Its proposed fix is a new aggregate layer above Project (names are the report's; see the naming section):

- **Program** — a lifecycle aggregate over several explicitly linked Projects, stored in its own Git-backable *Program workspace* (not inside any one repo, not hidden in `$ORGASMIC_HOME`).
- **Initiative** — the primary lifecycle object inside a Program: owns scope, affected Projects, decisions, risks, tasks, agent runs, and evidence. Task stays the execution unit; Project stays the repo boundary.
- **Typed Org event log** — append-only write authority for all Program/Initiative changes; human-readable Org headings with typed JSON payloads.
- **Rebuildable projections** — generated current-state views (`initiatives.org` etc.); stale ones are deleted and rebuilt, never hand-edited.
- **Stage + Health + Gates** instead of one status — `In Review / Blocked` becomes a valid, informative state.
- **Evidence Pack** — a first-class reviewable proof bundle that closes a gate ("task Done ≠ release readiness; agent claim ≠ accepted outcome").
- **Waiver** — an audited exception event that lets an operator pass a gate anyway, leaving a visible risk trail.
- **Agents as audited automation** — not "team members": every assignment records intent, scope, allowed paths, commands, claimed outcome, reviewer verdict.
- **External systems as adapters** — GitHub/CI/Jira import signals and evidence but never own lifecycle truth.
</RichText>

</Callout>

<Section title="Reusable even without the superstructure">

<RichText>
If the team rejects Program/Initiative wholesale, four ideas still stand on their own and could land as small, local changes:

1. **Stage + health instead of one status** — even at Task or goal level, `In Review / Blocked` is more honest than one Kanban column.
2. **Evidence-linked done-ness** — "an agent claimed success" ≠ "accepted outcome". A lightweight required-evidence convention on task close (link a run, a review verdict, a command output) captures most of Evidence Pack's value without a new object.
3. **Typed events over freeform notes** — wherever we log durable facts, the typed-Org-heading + JSON-payload pattern is stronger than prose, and matches our existing tx philosophy.
4. **Agents as audited automation** — already how orgasmic treats workers/runs; the report merely names the stance. Worth a glossary term regardless.
</RichText>

</Section>
