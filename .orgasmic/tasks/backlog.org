* BACKLOG TASK-3K9ZG Sanitize artifact ids on submit + add artifact to id mint    :artifacts:daemon:security:hardening:
:PROPERTIES:
:ID: TASK-3K9ZG
:PRIORITY: high
:END:

** Description
Two coupled gaps in artifact id handling, one a security hole (found while verifying the 2026-07-06 orsl battle-test friction report, F6):

1. =post_artifact_submit= (crates/orgasmic-daemon/src/api.rs:9182) passes the caller-supplied =art_id= path segment unsanitized into =artifact_dir()='s plain =join= (crates/orgasmic-daemon/src/artifacts.rs:105). No =ART-= shape check, no traversal guard — a crafted id can escape the artifacts dir. Same applies to every artifact route taking =Path(art_id)=.
2. Artifacts are the one node type outside the minting system: =orgasmic id mint --class artifact= is rejected, yet =artifact submit= demands a caller-invented id. Fresh ids exist server-side (=new_artifact_id=, artifacts.rs:661, ART- + 5-char Crockford) but are unreachable from the CLI.

Fix: validate id shape (the =new_artifact_id= grammar) on all artifact routes, rejecting traversal with an error that names the accepted shape; add =artifact= to =id mint --class= (unify =new_artifact_id= with =NodeIdClass= in orgasmic-core/src/id.rs) or mint-and-print when the id is omitted on first submit; document the accepted shape in =artifact submit --help=.

** Acceptance Criteria
- [ ] Traversal / malformed ids rejected on every artifact route, with a test proving an escape attempt 400s and writes nothing.
- [ ] A lawful way to obtain a fresh artifact id from the CLI (mint class or optional-id submit that prints the minted id).
- [ ] =artifact submit --help= documents the accepted id shape.
- [ ] Error message names the fix (accepted shape + how to mint), house style.

* BACKLOG TASK-JJ9RD Config node write path + kind registry single-source (silent wrong-file write)    :daemon:cli:hardening:
:PROPERTIES:
:ID: TASK-JJ9RD
:PRIORITY: high
:END:

** Description
Worst bug of the 2026-07-06 orsl battle-test (friction report F7+F8): =project.org= (=* PROJECT=) and =config.org= (=* CONFIG=) both scaffold =:ID: {{PROJECT_ID}}=, and the daemon's =NodeLayer= enum (crates/orgasmic-daemon/src/api.rs:6883) has no =Config= variant. Consequences: config.org has zero CLI write surface (violating the "no hand-edits" convention — an agent filling TEST_CMD/LINT_CMD/BUILD_CMD has no lawful path), and the natural attempt =node prop set <project> TEST_CMD ... --kind project= silently succeeds against the wrong file (project.org). Separately, the CLI =--kind= flag is a free-form =Option<String>= (crates/orgasmic-cli/src/node.rs:32) — no enum in =--help= at all; the accepted set lives only in =NodeLayer::from_kind= (api.rs:6953), which under-documents =handoff=/=goal= and cannot express =config=.

Fix:
- Add =NodeLayer::Config= with kind-scoped id resolution (kind+id → file via =org_node_path=, api.rs:7468, routing to config.org). Kind-scoping avoids migrating existing projects' files.
- Reject CONFIG-reserved keys (TEST_CMD/LINT_CMD/BUILD_CMD/PIPELINE) under =--kind project= with an error naming =--kind config=.
- Single-source the kind registry (expose from a shared location) so the CLI derives a real clap enum: =--help= lists exactly the daemon-accepted kinds, including =handoff=, =goal=, =config=.

** Acceptance Criteria
- [ ] =node prop set --kind config= / =node body set --kind config= write config.org; round-trip test.
- [ ] CONFIG-reserved key under =--kind project= → 400 naming =--kind config= (regression test for the silent wrong-file write).
- [ ] CLI =--kind= help enumeration generated from / parity-tested against the daemon registry.
- [ ] Existing projects (shared =:ID:=) work without file migration.

:ID: TASK-4T80X
:PRIORITY: high
:END:

** Description
Battle-test F4: =glossary create --relates-to "Semantic Routing Model, Expanded Routing Graph"= (prose, not ids) was accepted at write time, then poisoned the whole index — one dangling-ref parse error per whitespace-separated word, surfaced only as an opaque global counter, root-causable only by grepping daemon logs. Reference-valued properties are validated exclusively at index time (=lint_dangling_graph_edges=, crates/orgasmic-daemon/src/index.rs:1060; =collect_dangling_reference_tokens=, orgasmic-core/src/identity_lint.rs:184); =render_graph_heading= (api.rs:7850) and =node prop set= write property values verbatim.

Fix: validate ref-valued properties (RELATES_TO, GLOSSARY_REFS, MOTIVATED_BY, DEPENDS_ON, IMPLEMENTS, PARENT — the dec_HJENQ vocabulary) at write time in graph create handlers and =node prop set=: every token must resolve against the index snapshot. Reject with an error naming the property, the offending token, and the expected format (space-separated node ids); =--force= escape hatch for intentional forward references. Document the value format in the =--relates-to= / =--property= help text. Write-time rejection costs one retry; index-time poisoning cost a debugging session.

** Acceptance Criteria
- [ ] Create/prop-set with an unresolvable ref token → 400 naming property + token + expected format; nothing written.
- [ ] =--force= writes anyway (dangling lint still catches it later).
- [ ] Help documents the space-separated-ids format.
- [ ] Regression test reproducing the battle-test prose case end to end.
