Model/effort: orgasmic carries ZERO model knowledge (dec_WDR5K item 9). Structured catalogs come from PROTOCOL-LEVEL adapters only — the existing runtime_options_catalog trait method: codex (app-server RPC) and hermes (inventory capabilities) already implement it; probe whether claude ACP and cursor-agent ACP report models at session/new (cursor docs https://cursor.com/docs/cli/acp document initialize/authenticate/session/new/session/prompt and an optional model string on cursor/task but NO discovery method — verify against the live binary, do not assume). Boundary rule: adapters parse structured protocol responses, NEVER CLI text output — text parsing is the model-catalog equivalent of scrollback scraping. Harnesses without a protocol catalog degrade to a free-text picker with the harness default as placeholder; the manager may read harness CLIs as judgment but no orgasmic machinery depends on parsing free-form output. No provider-level static fallback, no :MODELS:/:REASONING_EFFORTS: reads, no validation of overrides: dispatch without explicit model passes no model flag (harness default applies); explicit override passes through verbatim and a nonexistent model is the harness's error, surfaced to the manager.
