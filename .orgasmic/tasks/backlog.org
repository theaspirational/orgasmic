* BACKLOG TASK-P8ZSP Native Windows PATH support for the CLI (PowerShell/cmd, %USERPROFILE%)    :install:path:windows:platform:
:PROPERTIES:
:ID: TASK-P8ZSP
:END:

Extends dec_CKMW3 and dec_YKFZX. The current PATH-hardening is a POSIX-shell mechanism (env file sourced from ~/.*rc); native Windows shells do not read it. Make the CLI reachable on native Windows.

Work required:
- Home resolution: orgasmic-core Home::from_env uses $HOME; add %USERPROFILE% (and/or %ORGASMIC_HOME%) fallback for Windows.
- Installer: install.sh is bash-only; provide a Windows install path (install.ps1 or equivalent) that downloads/places the runtime bundle.
- PATH wiring: instead of a sourced sh env file, set the user PATH via PowerShell $PROFILE and/or setx / the User-PATH registry value; make it idempotent and reversible.
- Binary linking: Windows symlinks need Developer Mode/privilege — fall back to copying the binary or a shim .cmd/.exe launcher into a bin dir on PATH.
- CLI surfaces: path_env::relink_source_binary currently bails on non-unix; add Windows variants for `orgasmic path ensure`, `orgasmic doctor --fix`, and the doctor 'cli on PATH' finding.
- Verification: confirm `orgasmic` resolves in a fresh PowerShell and cmd session.

Note: WSL and Git-Bash/MSYS2 already work via the unix path; this task is specifically native PowerShell/cmd.

* BACKLOG TASK-KNENN Harden default instructions: route implementation through orgasmic workers, not raw provider CLIs    :conventions:agent-dx:
:PROPERTIES:
:ID: TASK-KNENN
:END:

Harden the agent-facing default instructions so the next agent dispatches implementation work through orgasmic workers, never a raw provider CLI.

Observed failure: asked to "implement all, use codex 5.5 xhigh on fast" inside this managed repo, an agent reached for the raw `codex` binary (`codex --help`, `codex exec ...`) instead of `orgasmic manager dispatch`. Shelling out to a provider CLI bypasses the daemon write authority, tx recording, worktree isolation, dispatch leases, and reattach/recovery — the entire reason work is managed here.

Fix: add an explicit rule to `conventions/orgasmic-tooling.org` (and/or `contributing.org`) stating that any request to implement/dispatch work — especially one naming a provider+model+effort like "codex 5.5 xhigh" — maps to `orgasmic manager dispatch --kind implementer --worker implementer-<harness>-<driver> --model <model> --effort <effort> --brief <path>`, resolving the worker from `orgasmic worker list` (e.g. codex 5.5 xhigh -> implementer-codex-tmux, model gpt-5.5, effort xhigh; service_tier fast comes from the codex harness config). Raw `codex`/`claude`/`cursor-agent` invocations are off-limits for doing project work. State the why (write authority, tx, leases, reattach) so it sticks.

Scope: docs/conventions only; a worded rule + a short mapping table from "provider model effort" to worker id. No code. Consider whether the daemon should also surface workers more discoverably to agents.
