* Convention: contributing changes

How to make and record changes in this repo so the durable state stays the
source of truth.

- Follow the constraints in `project.org`.
- Keep personal data out of durable state: `tx/`, `decisions.org`, and task evidence are committed and may be published. Use a stable handle (e.g. a GitHub username), not a personal email, in `:ACTOR:`/`:SET_BY:` fields; write paths relative to the repo or as `$ORGASMIC_HOME`/`~`, never absolute home paths; don't record machine hostnames or other environment identifiers. Machine-specific notes belong in the gitignored `tmp/local_instructions.org`.
- Record durable rationale in `decisions.org`, never as stray markdown docs.
- While the project is greenfield (pre-stable), the decision and architecture graph is a *living model*: edit nodes in place to keep them true. Use `orgasmic decision create` / `orgasmic architecture create` for genuinely new rationale, and `orgasmic node body set` / `orgasmic node prop set` to revise, merge, correct, or re-link existing nodes. Reserve append-only supersession (a new record plus a `:SUPERSEDES:` backref, per [[dec_KTF04]]) for after the graph stabilizes, when history must be preserved. Edge keys come from the canonical vocabulary ([[dec_HJENQ]]): `:MOTIVATED_BY: :RELATES_TO: :DEPENDS_ON: :BLOCKED_BY: :IMPLEMENTS: :PRODUCES: :GLOSSARY_REFS: :PARENT:` — don't invent new ones (`:PARENT:` is the single-value, same-class decision-nesting hierarchy edge, per [[dec_2GQ7Q]]).
- Implement/dispatch work through orgasmic workers, never a raw provider CLI (`codex`, `claude`, `cursor-agent`) — see the "dispatching implementation work" convention in `orgasmic-tooling.org`. Raw CLIs bypass daemon write authority, tx, leases, and reattach.
- If something about the project surprises you, add an entry to `gotchas.org`.
- When you change a task's scope, keep `tasks/` current.
- To resolve any pre-migration id: =git show id-migration | grep <old-id>=.
