** Per-node test mapping (:TESTS:)

Each leaf node (=arch_NNN.M=) carries a =:TESTS:= property naming the *minimum* test commands an agent should run when it touches that node's =:SOURCE_PATHS:=. The goal is to avoid running the whole workspace suite for a change scoped to one module: run the node's =:TESTS:= (plus any node whose =:SOURCE_PATHS:= you also edited), not =cargo test --workspace=.

Conventions:
- The value is one or more shell commands, =;=-separated, runnable from the repo root.
- Rust commands are crate-scoped: =cargo test -p <crate>= runs that crate's lib unit tests *and* its integration test binaries. The mapping is by *crate ownership* of the source paths, which is stable against test-file renames.
- UI nodes (=ui/...= source paths) have no JS test runner in v0.0.1; their gate is =cd ui && npm run typecheck= (tsc).
- A node whose sources span two crates lists a command per crate.
- Run =cargo clippy -p <crate>= and the relevant =:TESTS:= before handing off; the full =cargo test --workspace= is a pre-merge / CI gate, not a per-change gate.
- Crate → test target reference: =orgasmic-core= (lib + =tests/fixtures.rs=); =orgasmic-drivers= (lib + =tests/driver_modes.rs=, =tests/cursor_agent.rs=); =orgasmic-daemon= (lib + =tests/integration.rs=, =tests/dispatch_endpoint.rs=); =orgasmic-cli= (lib + =tests/{dispatch,cli_parity,doctor,status,init}.rs=).
- When you add or move a node's =:SOURCE_PATHS:= across a crate boundary, update its =:TESTS:= to match.

** Purpose
Own the skill-first install and connection boundary: CLI-managed local daemon,
daemon-served browser UI, thin host app launchers, remote daemon profiles,
mobile access, contributor source fallback, user-authored overrides, runtime
state, sessions, secrets, logs, and executable entrypoint.

** Interfaces
- The only public install front door is the =/orgasmic= skill: install the
  skill first, then run =/orgasmic install=. The skill wizard recommends and
  executes the selected runtime path.
- CLI/runtime install builds or refreshes =$ORGASMIC_HOME/bin/orgasmic=,
  embeds shipped content plus the daemon-served web UI, creates
  =$ORGASMIC_HOME=, and manages the local daemon lifecycle with
  =orgasmic daemon start|stop|restart|status= and auto-starting daemon-backed
  commands.
- Host desktop apps are clients, not private runtimes. The Apple Silicon macOS
  app is a thin Tauri launcher that discovers the installed CLI/runtime, asks it
  for a one-time daemon UI launch URL, and opens that UI in the native webview.
  It may require one-time Gatekeeper approval while distributed as an
  ad-hoc-signed tester build.
- Remote Linux daemon setup installs the daemon on the Linux host, supervises it
  with =systemd --user=, keeps the default bind on =127.0.0.1=, and connects
  from macOS through an SSH tunnel or trusted private network.
- Mobile access is client-only: Android can use the Android app when release
  assets are available, while iOS or Android users who do not want the app use
  the daemon-served =/app/= UI through an HTTPS tunnel with an outer auth layer.
- Contributor-mode installer and updater create or refresh
  =$ORGASMIC_HOME/orgasmic=, build the binary, and maintain
  =$ORGASMIC_HOME/bin/orgasmic=.
- Installer uses SSH-first clone, HTTPS fallback, optional =--branch=, =--dir=, and =--no-build= flags, and scaffolds the home directories.
- =orgasmic update= fetches, checks out the selected branch, pulls fast-forward-only, builds release, refreshes the symlink, and auto-stashes/restores local source-checkout edits.
- Loader resolves =user/<relative-path>= before shipped content, with optional and hub content installed explicitly.
- Config, secrets, logs, sessions, and snapshots are addressed through paths under =ORGASMIC_HOME=.

** Constraints
- Default home is =~/.orgasmic= and remains overridable by =ORGASMIC_HOME=.
- User customization must not mutate shipped files in =orgasmic/=.
- Skill-only project scaffolding must not require Rust, Node.js, npm, a source
  checkout, daemon, app, or manually started dev server.
- Runtime source builds require Rust plus Node/npm because the CLI embeds the
  daemon-served UI.
- A host app must not spawn or bundle a private daemon that conflicts with the
  CLI-owned daemon lifecycle.
- The daemon manages projects on the machine where that daemon runs; a remote
  daemon cannot manage Mac-local paths.
- Remote daemon autostart belongs to the remote host service manager, not a
  macOS LaunchAgent.
- Mobile browser access must not expose a raw daemon; keep the daemon bound to
  localhost and put HTTPS plus outer auth in front of it.
- v0.0.1 keeps secrets as gitignored plaintext files and emits no telemetry.

contributor source path.

usable from a browser, Tauri desktop shell, Android client, or tunnelled mobile
browser. Host apps discover and reference that runtime instead of packaging an
independent daemon.

- Prompt templates are keyed by =(kind, state)= and provide spawn-time and continuation injections.
- Convention bundles are agent-facing behavioral rule documents at =shipped/conventions/<name>.org= (overridable at =user/conventions/<name>.org=) addressed from prompts via the slot =={{conventions.<name>}}==; the compiler injects the bundle's body verbatim wherever the slot appears.
- Slot compilation performs strict variable substitution with no conditionals or loops; dry-run reports unknown or unfilled slots. The slot registry includes the =skills.*=, =skill_path.*=, and =conventions.*= namespaces alongside the static =task.*=, =project.*=, =worker.*=, =run.*=, =evidence.*=, and =worklog.*= sets.
- Skill injection supports manifest, explicit full-body, and skill-path variants, with the manifest as the default prompt bundle behavior.
- The final prompt bundle is persona plus operating rules, skill manifest, applicable convention bundles, and the selected prompt injection handed to the driver.

** Constraints
- User overrides are loaded before shipped defaults for every overridable template, prompt, and convention bundle file.
- Missing linked skills and missing referenced convention bundles are compile errors in dry-run and block production spawn with a UI-visible error.
- Convention bundles carry no slot interpolation in their own bodies and are not template-engine inputs; they are inert behavioral text whose only structure is org headings and prose.
- The daemon does not enforce a bundle's content; bundles are advisory rules read by agents at spawn time, in the same epistemic class as the advisory in-code markers they may describe.
- =** Notes= sections (in agent templates, prompt templates, and convention bundles) are user-editable and are never injected into prompts.

* arch_ARSPJ Artifact store and versioning
:PROPERTIES:
:DEPENDS_ON: arch_BVH7M arch_C87Z9

** Description
A repo-local artifact is a folder: .orgasmic/artifacts/ART-<slug>/ holding artifact.mdx (current version, verbatim Agent-Native block vocabulary, opaque to the Org layer), artifact.org (index heading: :ID: :TITLE: :SUBJECT_NODES: :PROMPT: :VERSION: :STATE:), reviews.org (append-only comment headings: :CID: :AUTHOR: :VERSION: :ANCHOR: — PlanCommentAnchor JSON — :RESOLUTION_TARGET: :RESOLVED: :CONSUMED:), and versions/vN.mdx archives written at each regeneration. All writes go through the daemon writer and stamp tx entries: artifact.created, artifact.submitted, artifact.comment.added, artifact.comment.resolved, artifact.regenerated, artifact.generation.failed (the release-watcher revert when a generation run ends without a submit, arch_045Q0). The materialized index projects artifacts (list, per-node rollups, open-comment counts, regenerating state) from these files, and the event bus broadcasts artifact topics over WebSocket.

** Interface
- CLI (agent surface): orgasmic artifact blocks; orgasmic artifact submit <id> --file <mdx>; orgasmic artifact feedback <id> [--consume <cid>].
- CLI (admin): orgasmic member add/revoke/list (backing store is host-local, see the identity node).
- REST: GET /artifacts; GET /artifacts/:id[?version=N]; POST /artifacts/generate {nodes, prompt}; POST /artifacts/:id/regenerate {extraPrompt?}; POST /artifacts/:id/comments {anchor, message}.
- WS topics: artifact list/state changes, new comments.

** Constraints
- artifact.mdx is never parsed as Org and never hand-edited by agents; submit validates against the block registry before the writer accepts.
- Comments are append-only and version-scoped; regeneration archives the closing version's document and comments together (consumed), per dec_V44E4.
- The subject node-set is recorded explicitly at creation and does not track later graph edits.
- ART-<slug> ids follow the existing id style; the subject link back to nodes is index-projected, not stored on the nodes themselves.

** Description
A member layer over the daemon's existing bearer/session middleware. The single admin mints named tokens with a role (editor or viewer); records — name, role, token hash — live host-local in $ORGASMIC_HOME/user/auth/members.org, never in the repo. A new /login page accepts a pasted token and sets a long-lived named session cookie; the session check is extended to resolve member identity and role on every request. Comments and tx entries triggered by a member carry the member name.

** Interface
- CLI: orgasmic member add <name> --role editor|viewer (prints the token once); orgasmic member revoke <name>; orgasmic member list.
- HTTP: POST /login {token} -> named session cookie; all /artifacts/* routes read identity from the session.
- Role gates: POST /artifacts/generate and /:id/regenerate require admin|editor; POST /:id/comments requires any member; member management requires admin (the original daemon token).

** Constraints
- Tokens are stored hashed; the plaintext is shown exactly once at mint time and distributed out-of-band.
- Revocation invalidates all sessions minted from that token; re-minting under the same name preserves comment attribution (attribution is by name).
- Transport (tunnel, LAN, shared host) is deployment ops and out of scope, per dec_N9HW0.
- The pre-existing single daemon token remains the admin credential and the CLI's auth; member tokens are browser-facing only.
